# Design: chain — 採用 n8n 的 items 資料模型（取代 map/loop bolt-on）

Generated by /office-hours on 2026-06-04 (revised after n8n research)
Branch: feat/web-create
Repo: chain-cli
Status: APPROVED（設計方向）— 交給 /plan-eng-review 細化（地基級改動）
Mode: Builder
Scope: CLI / engine only（claude profile；UI 與 codex 之後）

## Decision（本次拍板）

**把 chain 的資料模型從「線上流一個字串」改成 n8n 的「線上流一串 items」。**
節點預設「對每個 item 各跑一次」。如此一來 loop 是隱性的，原本提的
`map` 節點與 `loop` 容器都作廢（見文末 Superseded）。

依據：n8n 官方「nodes run once for each item, so you don't need to build loops
into your workflows」。迭代是內建預設，不是節點。

## Core Model（對齊心智）

```
舊（現在）：線=一個值        節點跑一次
   node ──"東京..."──▶ node

新（n8n）：線=一串 items      節點對每個 item 各跑一次（自動）
   node ──[item,item,item]──▶ node     ← 自動跑 3 次，出 3 個 item

要「每城市先介紹再改推文」：不用迴圈、不用容器
   cities ─▶[Split Out]─▶[寫介紹]─▶[改推文]─▶[Aggregate]
   一個含陣列   拆成多      自動逐項   自動逐項    併回一個
   的 item     個 item     (隱性loop) (隱性loop)  含陣列的 item
```

## chain 的 item 定義（與 n8n 的關鍵差異）

n8n 的 item 是結構化 JSON；chain 的 ai 節點吐**文字**。所以定義：

```
Item = 一個 JSON 值（字串 / 物件 / 數字…）。線上流 Item[]。
ai 節點輸出：每個輸入 item 跑一次 → 該次的文字輸出 = 一個輸出 item。
  - 文字能 parse 成 JSON → item 值 = 該 JSON
  - 不能 → item 值 = 原始字串
root 節點（無上游）：跑一次 → 輸出 1 個 item（[item]）
```

所以一般線性鏈跟現在一模一樣（每節點 1 進 1 出 = 單元素 list），
**只有在出現多 item 時才會自動 fan-out**。向後相容。

## 節點清單（對齊 n8n）

```
節點            作用                                   對應舊概念
──────────────────────────────────────────────────────────────────
ai / cmd        對每個輸入 item 各跑一次（預設行為）      原 ai/cmd，但改成逐項
Split Out       一個含陣列的 item → 多個 item（fan-out）  (新) = 取代 map 的入口
Aggregate       多個 item → 一個含陣列的 item（fan-in）   取代 assemble 的合併角色
Merge           兩條輸入合併（append/by-key/by-position） 取代/補強 from:[a,b]
Loop Over Items 分批/限流/累積（少數情況才用）            (延後) 不是逐項處理用的
```

## 既有概念怎麼變

```
概念        現在                       items 模型下
──────────────────────────────────────────────────────────────
from: a     單上游                     不變：item 流從 a 進
$json       主上游整個輸出              改為「當前正在處理的 item」(n8n 語意)
$('id')     取某上游整個輸出            取該上游的 items（.all() 取全部；預設配對當前 item）
from:[a,b]  $json=a, 其餘 $('b')        多輸入改由 Merge 節點處理（語意更清楚）★ 見 Open Q
assemble    純資料模板合併              由 Aggregate 取代（items→一個含陣列 item）
$item/$index 我上版提的                作廢（n8n 用 $json=當前 item, 不需要）
快取         <id>.out 存字串            <id>.out 存「items 的 JSON 陣列」；節點仍是快取單位
CLI 顯示     ✓ node                     ✓ node (3 items)
```

## Migration 衝擊（誠實盤點，這是地基改動）

會動到的核心檔（待 eng-review 細化）：
```
types.ts     NodeResult.output: string → Item[]（或字串內裝 JSON 陣列）；新增 splitOut/aggregate/merge 型別
run.ts       runNode 改成「對輸入 items 逐項跑、收集成 items」；新節點各自的 items 轉換
render.ts    $json = 當前 item；$('id') 回傳 items；移除上版 $item/$index 構想
cache.ts     輸出序列化成 items 陣列；key 結構不變（prompt+type+upstreamKeys）
validate.ts  新節點的形狀檢查；多輸入/Merge 規則
cli/index.ts CLI 顯示 item 數
examples/*   demo / fan-in / test060316 全部要驗證在新模型下仍對（多半相容：1進1出）
所有測試      斷言從「字串輸出」改成「items 輸出」；新增 Split Out/Aggregate/Merge 矩陣
```

風險評等：**高**（全案最大）。但引擎還是 alpha，現在改比之後改便宜得多。

## Open Questions（給 eng-review）
1. **多輸入 vs Merge**：現有 `from:[a,b]` + `$('b')` 要保留，還是一律改用 Merge 節點？兩者並存會混淆。建議：`from` 單純表「誰餵我 items」；真正的合併語意（by-key/position）走 Merge。
2. **item 配對**：下游節點同時引用兩個上游時，item 怎麼對齊？（n8n 用 paired-item / by-position）需定義。
3. **cmd 節點**：逐項跑還是固定跑一次？（n8n code 節點可選 run-once / per-item）
4. **文字↔JSON**：item 值「能 parse 就當 JSON、不能就當字串」會不會有歧義？要不要明確 `parse:` 開關。
5. **Loop Over Items**（分批/累積）排在哪期？建議 v1 先不做。

## Success Criteria（可觀察）
- 一條 flow：node 產生 JSON 陣列 → Split Out → 下游 ai 自動逐項跑 → Aggregate 併回。CLI 顯示各節點 item 數。
- 單進單出的舊線性鏈（demo.yaml）在新模型下行為不變（向後相容）。
- 改某節點 prompt → 它 + 下游重跑，上游 cached（Merkle 不破）。
- 空 items → 下游不跑、輸出空、不報錯。
- 結構測試（validate / scaffold / ls）離線可跑；跑 ai 節點的 E2E 用真 `claude -p`、gated（已無 fake model）。

## Locked Decisions（/plan-eng-review + codex 外部審查 2026-06-04）

- **範圍**：一次做完整個 items 模型。D1=B。
- **item 形狀（修正後）**：**n8n 信封 `{json: value, pairedItem?}`**，節點輸出 `Item[]`。pairedItem 記錄來源項，讓配對/Merge/`$('id')` 有依據。D2 原裸值已**推翻**，D4=C。
- **多輸入**：兩條獨立輸入流要合併 → **Merge 節點**（n8n input-2）。`$('id')` 引用任一**祖先**節點，靠 pairedItem 解析配對項（n8n `$('Node').item`），算 DAG 邊、計入 topo/快取。兩者不再衝突（pairedItem 給了 item 身分）。原 D3 的不連貫由 C 解掉。**仍會打掉現有 fan-in.yaml + 多輸入 e2e，需遷移。**
- **集合運算子（codex 修正）**：Split Out / Aggregate / Merge 是**集合運算子，看整串陣列**，不是逐項管線。逐項管線只用在 ai/cmd(perItem)。**不要**把它們塞進同一條逐項迴圈。
- **Aggregate 與空（codex 修正）**：Aggregate 是集合運算子，**不適用**「空→跳下游」；空輸入時 Aggregate 仍跑、吐一個含 `[]` 的 item。
- **cmd（codex 修正）**：加 `mode: once | perItem`，預設 `once`。Split Out 後要逐項處理就用 `perItem`，避免 N items 被默默壓成 1。
- **保留 assemble（codex 修正）**：assemble 是純模板排版（多上游→一份文件），**Aggregate 不是它的替代**，兩者並存。
- **快取**：ENGINE_VERSION 1→2（舊 string 格式全失效）；`CacheStore` 序列化 `Item[]` 為 JSON、load 回傳 `Item[]`（型別邊界明確，不留「string 或 Item[]」模糊）；新節點設定（split 欄位 / aggregate 欄位 / merge 模式 / cmd mode）**必須**折入 computeKeys。node 級快取＝改一個上游 item 會重跑全部下游 item 呼叫（v1 接受，逐項快取延後）。
- **不 auto-parse（codex 修正）**：ai 輸出的 item.json = 原始文字（不靜默 parse，避免 markdown fence 等誤判）；要結構化用 Split Out / 顯式 parse。
- **render（codex 修正：不是小改名）**：`$json` = 當前 item.json；`$('id').item`（靠 pairedItem）/`.all()`；補根層 `$json[*]`。是一塊實打實的 render 重寫。
- **Merge 規格**：append / by-key / by-position 各自的輸出形狀、key path、重複鍵、缺鍵、長度不一行為都要明定（不能只寫「三模式」）。
- **節點機器名鎖定**：`splitOut` / `aggregate` / `merge`（小寫，同 ai/cmd/assemble）。
- **UI 不是真的延後（codex 修正）**：`cli/index.ts` import web server、web server import engine types/render/Runner；改 `NodeResult.output` 會弄壞 web server build。遷移**必須**同步更新 `src/web/server.ts`（至少編得過），UI 的 UX 才延後。
- **失敗語意**：逐項中某項失敗 → 整節點 failed、先前已跑的項丟棄、重試重跑全部（v1 刻意如此；逐項 checkpoint 延後）。
- **plan/preflight**：`planRun`/`aiCallCount` 對 fan-out 節點特判；materialize 前顯示 `×?`，之後給實數。
- **不做平行**：逐項序列跑 N 次子行程。
- **測試政策（使用者要求）**：每個 CLI 情境/節點都要有一個 **CLI E2E**（vitest、`e2eCli/scenarios/*.e2e.ts`），與瀏覽器測試（Playwright、`e2e/browser/*.spec.ts`）**完全分離**。**無 fake model**：跑 ai 節點的 scenario 用真 `claude -p`、gated（無 claude 就 skip）；結構測試離線。新能力沒配 CLI E2E 不算完成。

## NOT in scope
- **Loop Over Items（分批/限流/累積）**：少數情況才需要，延後。
- **item 平行化**：逐項仍序列跑；平行是獨立議題。
- **{json} 包裝 / binary / pairedItem metadata**：YAGNI，裸值現在夠。
- **UI 呈現 items / Split Out / Aggregate**：UI 之後另談。
- **codex profile**：本案只 claude。

## What already exists（重用，別重造）
- `render.ts` 的 JSON path 選擇器（`$json[*]` 等）→ Split Out / item 取值直接沿用。
- `plan.ts` plan/execute 分離 → 逐項的 aiCallCount 在上游 materialize 後才算得出（loop `×?` 那題）。
- `cache.ts` Merkle 機制 → 沿用，只改序列化 + 折入新節點設定 + bump 版本。
- `$('id')` 引用 → 保留為跨上游引用（n8n 也有）。
- fast-fail (E2)、scratch/--pin、FlowLock → 不變。

## Failure modes（每條新路一個真實production失效）
- 逐項中某項 claude 失敗 → 須整節點 failed + 下游 skip（有測試 T1）。否則部分成功會留半套 items（靜默壞資料）。
- 上游非陣列卻接 Split Out → 必須報清楚錯（有測試 T4），否則靜默當單 item。
- 改 Merge 模式但 key 沒折入 → 靜默舊快取（★critical，T3 回歸測試守）。
- 舊 .chain 快取是 string 格式，新引擎讀成 items → ENGINE_VERSION bump 讓它整批失效（T3）。

## Superseded
本文件取代同檔上一版的 `map` 節點 + `loop` 容器 bolt-on 設計。原因：那是在「單字串模型」上硬貼迴圈，與 n8n 思路不符；改用 items 模型後，逐項處理是隱性預設，map/loop 容器不再需要。
