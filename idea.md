# chain 重做 — 設計交接文件（高層次指引）

> 給接手的設計／開發者：這份文件提供**方向與邊界**，不規定實作細節。
> 詳細的資料結構、介面、模組切分由你設計；本文界定「要解決什麼、為什麼這樣選、哪裡是地雷」。

---

## 0. 一句話定義

> 讓使用者用**一份設定檔**，把「要一步步餵給 AI 的多個提示」串成一條**可重跑的流程**，跑完自動把每步結果存好。

設定檔是主角；CLI 與 UI 只是它的兩個消費者。

```
                    ┌───────────────────────────────────┐
                    │      設定檔（單一真實來源 SoT）      │
                    │  = 提示詞 + 迴圈 + 輸入/輸出接線     │
                    │  以一份 .yaml 存在專案某處           │
                    └───────────────┬───────────────────┘
                讀／視覺化／編輯 │          │ 讀取後反覆執行
                       ┌───────────┘          └───────────┐
                       ▼                                  ▼
              ┌──────────────────────┐         ┌──────────────────────┐
              │  網頁 UI（給「人」看） │         │  CLI（給「跑」用）     │
              │ 打開就看懂整條流程     │         │ 一鍵跑、可接續、可彙整 │
              │ 就地編輯、逐站推進     │         │ 同一份設定產一致結果   │
              └──────────────────────┘         └──────────────────────┘
```

## 1. 設計北極星（決定成敗的不是「能跑多步」）

「能跑多步」是基本盤。真正決定好不好用的是兩組體驗，請把它們當第一優先：

```
B 系列 — 迭代體驗   釘樣本只跑這步、改一步只重跑受影響的部分（不重跑昂貴上游）
E 系列 — 排錯接續   失敗不丟已完成成果、改完能接續、錯誤可行動
```

特別注意：本工具的 AI 步是用**本機 CLI 大模型**（如 `claude -p`、`codex -m`）跑的，**慢且耗真實額度**。所以「能跳過就跳過、能只重跑受影響的就別全跑」不是錦上添花，而是核心價值。一切快取／接續設計都圍繞這點。

## 2. 技術棧與全域原則

```
相依      可有相依，但求少而精、好搬遷（已放寬原始的「零相依」要求）
語言      TypeScript，開發期用 tsx 直跑（dev 免 build；發佈可選擇性編譯）
設定檔    YAML，唯一真實來源；只存「邏輯圖」，保持人類可讀
          （座標、佈局等視覺資料不進主檔；UI 自動佈局，必要時另存 layout 邊檔）
入口      單一 binary `chain` + 子指令（run / validate / ui / ls）
```

全域鐵則：

- **設定檔可讀優先**：使用者的第一情境是「過陣子打開設定檔看懂整條流程」。任何會讓 YAML 變雜的東西（座標、冗長 schema、內嵌大段資料）都要擋在主檔外。
- **同一引擎、同一驗證器**：CLI 與 UI 共用同一套執行引擎與設定驗證器，絕不各寫一份。
- **誠實回報（G3）**：跳過、失敗、迴圈逐筆成敗，都要如實標示，不得假裝成功。

## 3. 設定檔定位與目錄佈局

```
一流程一檔；一個專案裡會有「多份」設定檔散落各資料夾。
CLI / UI 都以「路徑」指定要跑/看的那一份；UI 額外可掃描專案列出清單（chain ls）。
設定檔內的相對路徑（input/output）以「執行時 cwd」為基準。
.chain/ 暫存與狀態，放在「設定檔」旁邊。
```

地雷與防護：cwd 基準會讓「同一份設定從不同目錄跑 → 讀寫到不同檔」。**CLI/UI 啟動時一律印出解析後的 cwd 與每個 input/output 的絕對路徑**，讓漂移當場可見。`.chain/` 跟著設定檔走（不是跟 cwd），避免狀態散落。

目錄長相：

```
你的專案/
├─ my-flow.yaml              ← 設定檔（唯一真相）
└─ .chain/                   ← 跟在設定檔旁
   ├─ outputs/               ← 每步正式暫存結果（讓「沿用/接續/只重跑下游」成立）
   │   ├─ fetch.txt
   │   └─ tweets.json
   ├─ scratch/               ← 試跑（--pin）的隔離輸出，永不污染 outputs/
   └─ state.json             ← 每步「跑過沒 + 當時提示詞 hash + 狀態 + 輸出路徑」
```

## 4. 流程骨架（n8n 心智模型）

UI 的觀感與互動以 n8n 為參照：節點 + 連線 + 節點內用運算式挑欄位。但**執行語意**是我們自己定的（見第 6 節）。

### 4.1 節點型別（ai 是一等公民，是整個系統存在的理由）

```
┌──────────────────────────────┐
│ ★ ai 步（主角）               │ 用 CLI 大模型；專屬：schema+重試(C4)、換 profile(B3)
│                              │ G2 假模型；登入態。系統就是為它而生。
├──────────────────────────────┤
│   cmd 步（次要）              │ 跑任意 shell 指令（取資料、後處理…）
├──────────────────────────────┤
│   assemble 步（純資料）       │ 只做取值/組裝/運算，不呼叫外部（對應原文 C1 純組裝）
└──────────────────────────────┘
      底層：ai 與 cmd 共用一個 subprocess runner（省得寫兩份）
```

### 4.2 接線與運算式

```
線(from)   決定「這個節點看得到誰的資料」；也就是依賴邊。
$json      = from 指到的那個上游的輸出。
$node["X"] = 指名另一個上游（多輸入時用）。
迴圈內 $json = 當前這一筆 item。
```

線**顯式寫在 YAML**（一行 `from:`）。改提示詞不會動到拓樸——這正是 n8n 好用之處，也讓「線歸線、文字歸文字」一目了然。

運算式分兩層（這個分層讓開跑前驗證能成立）：

```
第一層 路徑選擇器（不算 JS，可靜態驗證）── 主力
  $json.title              取欄位
  $json.items[2].text      取陣列第 N 筆的某欄位
  $json.items[-1].text     取最後一筆
  $json.items[*].title     整欄抽成清單（pluck）
  $node["trend"].score     指名上游的欄位

第二層 唯讀沙箱（真的要算才用，少見）
  {{ $json.a + $json.b }}、{{ $json.t.toUpperCase() }}、三元
  禁止 require / 讀寫檔 / 網路 —— 那些副作用請改用「cmd 步」明確地做
```

> 取捨備忘：第二層沙箱碰到的欄位無法靜態確定，因此「打錯欄位名」在沙箱運算式裡只能跑時才炸（歸 E1 報錯）；第一層路徑選擇器則可在開跑前擋下。預設鼓勵用第一層。

### 4.3 迴圈（容器節點）

```
迴圈 = 一個「容器節點」，內含自己的子流程（可多步、子步間可接力）。
over: 必須解析成一個 list；as: 把每筆綁成 item。
執行：嚴格逐筆（跑完一筆才下一筆），輸出保序。
空清單 over=[] → 輸出 []（不算失敗）。
某筆失敗 → 快速失敗 + 保留已成功筆（停下游）；continueOnFail 可當每迴圈可選旗標。
輸出：整包清單 {{loop}}；可 pluck 抽單欄 {{loop[*].title}}。
巢狀允許，但內外層 as 別名須不同（MVP 可先只測一層）。
```

### 4.4 設定檔示意（僅供對焦，非定稿；實際 schema 由你設計）

```yaml
profiles:
  default: { cmd: 'claude -p' }            # prompt 走 stdin，不塞進引號
  fast:    { cmd: 'claude --model haiku -p' }
  codex:   { cmd: 'codex -m gpt-5' }

steps:
  fetch:
    type: cmd
    run: 'cat ./input.txt'                 # 相對路徑以 cwd 為基準

  summarize:
    type: ai                               # 預設用 default profile
    from: fetch
    prompt: "抽出三個重點：{{ $json }}"

  tweets:
    type: ai
    from: summarize
    prompt: "依重點產 10 則推文，回 JSON 陣列"
    schema: { items: [ { text: string } ] }   # C4：可選，宣告了才驗

  title_each:                              # 迴圈容器
    over: "{{ tweets.items }}"
    as: item
    steps:
      title:
        type: ai
        prompt: "為這篇下個標題：{{ item.text }}"
        # 每筆收斂成 { title: ... }

  report:
    type: write                            # 寫成品節點（見第 8 節）
    from: title_each
    mode: append
    path: "reports/{{date}}.md"
```

## 5. 模型呼叫（CLI 子行程，不是 API）

不走 HTTP API、不需金鑰，靠本機已登入的 CLI。`ai` 步透過 profile 決定指令樣板。

```
B3 換模型   = 換 profile（CLI `--profile fast` 一鍵全換；個別步可自寫覆蓋）
G2 假模型   = profile 換成 echo / cat fixture，引擎一行不改即可離線自我測試
```

子行程鐵則（API 沒有、CLI 才有的坑）：

```
① prompt 一律經 stdin 餵給子行程，不塞命令列 → 免長度上限、免跳脫地獄
② 用 argv 陣列 spawn（不經 shell 字串）→ 免注入；cmd 步要用 shell 時明確標記
③ 每個 ai/cmd 步設 timeout，逾時強制 kill
④ 失敗訊息：把 stderr 原文帶進 E1 報錯（如登入態過期、command not found）
⑤ 開跑前 which 檢查指令存在性（見 E4）
```

## 6. 執行模型（重要：視覺可並排，執行嚴格序列）

```
引擎：嚴格全序列，一次只跑一個節點；迴圈嚴格逐筆。
UI ：可把分岔/迴圈畫成「並排」或「迴圈框」——那只是視覺，幫人讀懂結構。
並排節點的實際執行先後 = 依 YAML 書寫順序（由上到下，可預測、結果不漂移）。
不做執行時並行（--concurrency 最多當未來實驗旗標，不列入預設設計）。
但引擎內部請以「拓樸排序」實作，讓未來要加並行時不必重寫。
```

視覺 vs 執行：

```
視覺上（UI，看起來並排）            執行時（引擎，嚴格序列）
      ┌─▶ [a1] ─┐                   a → a1 → a2 → a3 → merge → b
[a] ──┼─▶ [a2] ─┼──▶ [merge]──▶[b]   （依 YAML 書寫順序逐一跑，絕不同時開兩個）
      └─▶ [a3] ─┘
```

## 7. 資料 / 快取 / 接續（核心引擎）

這是整個工具最難、最值錢的一塊。三層資料各司其職：

```
① runtime       單次執行記憶體中的傳遞值（跑完即逝）
② .chain/ 暫存  工具「自動」存每步結果 → 讓沿用/接續/只重跑下游成立（B、E 系列引擎）
③ 成品檔        使用者在流程裡「明確接寫檔節點」輸出 → 給下游（網站/週報）吃（D 系列）
```

關鍵釐清：**暫存（工具自動，給迭代用）** 與 **成品（你明確寫，給下游用）** 是兩回事，不互斥。

「一個步驟算不算已完成」的判定 = **記錄「跑過 + 當時的提示詞（與輸入、profile）hash」**：

```
你改了 stepB 的 prompt ─▶ hash(B) 變 ─▶ state.json 比對不符 ─▶ B 失效
                                                            │ 沿 DAG 可達性
                                                            ▼
                                          B 的所有下游 C、D… 一併失效
run 結果：A=沿用(cached) → B=重跑 → C=重跑 → D=重跑；不相干上游全跳過（= B2）
```

執行行為（對齊使用者要的 n8n 體驗）：

```
情境1 整條跑         預設「沿用暫存」：沒改的步沿用、改過的步及其下游重跑
情境2 中途報錯       停在該點；保留前面已完成成果(E2)；可單獨重跑該點再往下
情境3 跑到某點/往後N步 點某節點「從頭跑到這」；或從目前往後 N 步
--fresh             忽略所有暫存，整條重頭跑
```

步驟狀態機：

```
PENDING
  │
  ├─ 有有效暫存？(該步 hash 未變 & 上游未變) ─是─▶ SKIPPED(cached) ─▶ 下游沿用其輸出
  否│
  ▼
RUNNING ──成功──▶ SUCCESS ─▶ 寫 .chain/outputs/ + 更新 state.json
  │
失敗/逾時/非0 exit
  ▼
FAILED ─▶ 保留已完成步輸出(E2) + 停下游 + 在 log 誠實標示(G3)
```

## 8. 周邊能力（圍繞核心長出來的）

### 8.1 B1 樣本試跑（迭代體驗第二條腿）

```
樣本獨立成檔（如 .chain/samples/tweets.json），試跑時 `--pin tweets=樣本檔` 指定。
試跑結果一律寫 .chain/scratch/ 隔離區，永不碰正式 outputs/ 或成品檔。
釘住的樣本當成該節點的「事實輸入」，下游可接著一路試跑（仍全寫 scratch）。
→ 讓你「只調受影響的那一小段」而不必重跑昂貴上游。
```

### 8.2 C4 結構化輸出

```
schema = 極簡欄位表（名 + 型別 + 陣列），寫在 YAML；可選，宣告了才驗。
流程：抽出 JSON（剝掉模型多嘴的前後廢話 / ```json``` 圍欄）
      → 比對 schema → 不符就「把差異回灌」讓模型修正、重試 1 次
      → 仍不符 = 該步 FAILED（訊息含：期望 X、實得 Y）。
注意：CLI 大模型沒有 API 的「強制 JSON 模式」，所以 prompt 要明確要求只回 JSON，
      且務必自己做抽取 + 驗證，不能假設它一定回乾淨 JSON。
深層約束需求現在不做；日後真要再升級成完整 JSON Schema。
```

### 8.3 D 成品輸出

```
「寫成品」= 一種節點（type: write），跟 n8n 接 Write 節點一致，UI 看得到、可接在任一分支末端。
mode: overwrite(D1 覆蓋) / append(D2 追加)。
內建日期變數 {{date}}（YYYY-MM-DD）、{{datetime}} 等，路徑與內容都能用。
鐵則：成品寫檔只在「整條流程成功」後執行（不寫一半）；試跑模式預設不寫成品。
append 同日去重：同一天/同識別碼已存在則「取代那段」而非重複追加（防例行手滑重跑）。
```

### 8.4 E4 開跑前驗證（在燒任何額度前擋下）

```
跑前「靜態」可驗（不花額度，全擋在開跑前）：
  • from 指到不存在的步、依賴成環、運算式引用沒接線的來源
  • 路徑選擇器指向上游 schema 沒有的欄位（該上游有宣告 schema 時）
  • profile 名打錯、ai 步的 CLI 指令不存在（which 檢查）
  • 迴圈 over 非 list、as 別名與外層衝突
報錯方式：
  • 「你是不是要打 X」用編輯距離猜最近的合法名 + 列出可用清單
  • 一次收集「全部」設定錯再一起報（不要遇到第一個就停）
共用：CLI 跑前驗、UI 存回設定檔前也驗；驗證不過 → 不落地（對齊 F4）。
```

## 9. CLI 介面

```
chain run <flow.yaml>          一鍵跑整條（預設沿用暫存）
        --fresh                忽略暫存，重頭跑
        --from <步>            從某步強制重跑 + 下游
        --to <步> / --steps N  跑到某步 / 往後 N 步
        --pin <步>=<樣本檔>    釘樣本試跑（寫 scratch）
        --profile <名>         換模型 profile（如 fast）
        --quiet                關閉即時串流
chain validate <flow.yaml>     只做開跑前驗證，不執行
chain ui <flow.yaml>           啟動本機網頁編輯器
chain ls                       掃描專案、列出找到的 flow 設定
```

輸出風格（使用者選擇「不做結尾摘要，只看 log」）：

```
即時串流每步輸出（標步驟名）。不另做結尾總結表。
→ 因此「誠實」責任移進 log 本身：每步用顯眼前綴承載 E1/G3——
    ▶ tweets     開始
    ✓ tweets     完成 28s
    ⊘ fetch      跳過 (cached)              ← G3：跳過誠實標示
    ⟳ title #7/10 失敗：逾時                ← 迴圈逐筆當下就標
    ✗ merge      失敗：schema 缺 title       ← E1：壞在哪當場印
                 （並印「前面已完成：fetch, tweets」← E1 要的脈絡）
即使無總結，往回滑也能靠 ✓/⊘/✗ 前綴一眼掃到狀態。
```

## 10. UI 範圍（與 CLI 並行開發）

```
目標：一次做到 F1~F4。技術用「Node 內建 http + 原生前端」（畫布、連線、拖拉全手刻）。
共用：UI 讀同一份 YAML、呼叫同一引擎與驗證器。

F1 看每步全貌   左圖右詳：輸入實際值 / 原始提示 / 代入後提示 / 上次輸出
                左欄把 from 指到的上游攤平成欄位樹，點一下插入 {{$json.x}} 進提示
F2 就地改試跑   改提示 → 試跑（寫 scratch）→ 滿意才存回；存回前驗證，壞不落地
F3 逐站推進     一站站手動推進，或整條自動跑
F4 結構編輯     插入/刪除/調整步驟順序並存回 YAML；存前驗證，壞不落地
```

編輯三欄示意：

```
┌ 編輯某一步（n8n 式三欄）──────────────────────────────────────────┐
│ INPUT 上游送進來       │ PROMPT 編輯              │ OUTPUT 上次結果  │
│ ▾ $json (來自 summarize)│ 為這篇下標：             │ { "title":       │
│   • title  "AI 突破"    │ 「{{$json.summary}}」    │   "三分鐘讀懂…"  │
│   • summary "本文…" ───┼▶（點欄位→插入運算式）    │ }                │
│   • tags  ["AI","ML"]   │                          │                  │
└────────────────────────────────────────────────────────────────────┘
```
